## Context for Implementer

> This section is critical for cross-session continuity. Write it for an implementer who has never seen the codebase.

- **Patterns to follow:**
  - Repository tests: Use `sqlmock` with `sqlx.NewDb`. See `internal/repository/comment_repository_unit_test.go` for the standard pattern (setup mock, create repo, define expectations, assert).
  - Handler tests: Use `httptest.NewRequest` + `httptest.NewRecorder` with mock usecase interfaces. See `internal/httpapi/handlers/moderation/moderation_handlers_unit_test.go` for the established handler test pattern.
  - Table-driven tests are the standard: `[]struct{name, input, want}` with `t.Run`.
  - Use `testify/assert` for non-fatal checks, `testify/require` for fatal checks (e.g., `require.NoError` before proceeding).
  - All new test files use `*_unit_test.go` suffix for consistency.

- **Conventions:**
  - Test files: `*_unit_test.go` for new unit tests (consistency convention for Sprint 18)
  - Package: tests are in the same package (not `_test` suffix package)
  - Guard integration tests with `if testing.Short() { t.Skip(...) }`
  - Helper functions start with lowercase and use `t.Helper()`

- **Key files the implementer must read first:**
  - `internal/repository/comment_repository_unit_test.go` — canonical repo test pattern
  - `internal/httpapi/handlers/moderation/moderation_handlers_unit_test.go` — canonical handler test pattern
  - `internal/domain/errors.go` — sentinel errors used across tests
  - `scripts/coverage-thresholds.txt` — current coverage gates

- **Gotchas:**
  - Some repository files are large (redundancy: 815 lines, crypto: 562 lines) — focus on exported methods and error paths
  - **Handler dependencies may be concrete structs, not interfaces.** Before writing handler tests, check the handler struct's fields. If a dependency is a concrete struct instead of an interface, extract an interface from it first (minimal change: define the interface in the handler package, then mock it). This is expected scope.
  - The `httpapi/shared` package has 3 files (dependencies.go, helpers.go, response.go) totaling modest code — basic tests should suffice
  - `httpapi/compat` is an empty directory — ignore it
  - **Two distinct analytics handler files in video package:** `analytics_handlers.go` (live stream analytics, 416 lines, UNTESTED) and `video_analytics_handlers.go` (video analytics, has existing tests). Only the former needs new tests.
  - **OAuth handler tests (oauth.go, 740 lines)** may need session state handling (CSRF state, PKCE). Check whether the handler manages sessions via cookies or query params. If session-based, use `httptest.NewServer` with `cookiejar` instead of simple `httptest.NewRequest`. See `internal/httpapi/handlers/auth/auth_integration_test.go` for similar patterns.
  - **Federation hardening handler** includes HTTP signature verification and may call crypto libraries. See `internal/repository/activitypub_key_security_test.go` for how to generate test keys. May need test crypto fixtures.

- **Domain context:**
  - Repositories are SQLX-based, using `sqlx.DB` (not `*sql.DB`)
  - Handlers receive usecase interfaces via constructor injection
  - Error handling uses domain sentinel errors (ErrNotFound, ErrUnauthorized, etc.) that map to HTTP status codes
